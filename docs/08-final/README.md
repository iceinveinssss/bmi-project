## 1. Техническое задание

### 1.1 Общие сведения

**Наименование:** ИМТ Калькулятор — мобильное приложение для расчёта и мониторинга индекса массы тела.  
**Основание:** Курсовой проект, дисциплина «Программная инженерия», направление 09.03.04.  
**Сроки:** 01.03.2026 — 30.05.2026.

### 1.2 Функциональные требования

**Подсистема аутентификации**
- Регистрация: имя, email, пароль (мин. 6 символов), год рождения (необязательно)
- Вход с получением JWT-токена (срок 24 ч, claims: email, userId, role)
- Хранение токена: `SecureStore` на iOS/Android, `AsyncStorage` на web
- Выход: `authStorage.clear()` удаляет токен и профиль

**Подсистема расчёта ИМТ**
- Ввод веса и роста в метрических (кг/см) или имперских (lb/in) единицах
- Формула: `ИМТ = weight / (height/100)²`, округление до 0.1 (`BmiRecord.calculateBmi()`)
- 7 категорий ВОЗ (`BmiRecord.getCategory()`): от «Выраженный дефицит массы» до «Ожирение III степени»
- Сохранение на сервере: `POST /api/bmi/calculate`

**Подсистема мониторинга**
- История: `GET /api/bmi/history` → упорядочено по `measuredAt DESC`
- Фильтрация: `GET /api/bmi/search?category=...` (case-insensitive, partial match)
- Статистика: `GET /api/bmi/stats` → min, max, avg, count, currentCategory
- Редактирование: `PUT /api/bmi/{id}` (с owner-check через `findByIdAndUserId`)
- Удаление: `DELETE /api/bmi/{id}` (с owner-check)

**Подсистема оффлайн-режима**
- Кэш истории: `AsyncStorage`, ключ `bmi_history_cache_v1`
- Кэш статистики: `AsyncStorage`, ключ `bmi_stats_cache_v1`
- При сетевой ошибке — автоматическая загрузка из кэша
- Дата актуальности кэша отображается в `SettingsScreen`

**Подсистема настроек**
- Тема: Системная / Светлая / Тёмная (мгновенное применение)
- Единицы измерения: Метрические / Имперские
- Управление кэшем: просмотр статуса, кнопка «Очистить кэш»

**Административная подсистема**
- `GET /api/admin/users`: список всех пользователей
- Доступ только с ролью `ADMIN`; роль `USER` → 403 Forbidden

### 1.3 Требования к архитектуре

Система реализует PCMEF: C→M→E→F на сервере, адаптированный P на клиенте. Коммуникация C↔M через интерфейсы `IBmiService`, `IUserService`. Зависимости строго однонаправленные. Бизнес-методы расчёта ИМТ размещены в Entity (`BmiRecord`).

### 1.4 Требования к безопасности

- Пароли — только BCrypt-хеш (`BCryptPasswordEncoder`)
- JWT: HMAC-SHA, jjwt 0.11.5, `JwtFilter extends OncePerRequestFilter`
- CSRF отключён; сессии не используются (`STATELESS`)
- CORS: `allowedOriginPatterns("*")` (для разработки)
- Поля `password` и `bmiRecords` помечены `@JsonIgnore`

## 2. Руководство пользователя

### 2.1 Начало работы

**Установка Expo Go:**
- Android: [Google Play](https://play.google.com/store/apps/details?id=host.exp.exponent)
- iOS: [App Store](https://apps.apple.com/app/expo-go/id982107779)

**Запуск (после старта сервера):**
```bash
cd frontend
npm install
npx expo start
```
Отсканируйте QR-код в приложении Expo Go.

### 2.2 Регистрация и вход

1. Откройте приложение → экран «Вход».
2. Для новых пользователей: нажмите «Нет аккаунта? Зарегистрироваться».
3. Заполните: Имя, Email, Пароль (мин. 6 символов), Повтор пароля. Год рождения — необязательно.
4. Нажмите «Создать аккаунт» → автоматический вход.

### 2.3 Расчёт ИМТ

1. Вкладка **«Калькулятор»**.
2. Введите Вес и Рост (в единицах, выбранных в Настройках).
3. Нажмите **«Рассчитать»** → результат с цветовой индикацией категории.
4. Запись автоматически сохраняется в историю.

**7 категорий ИМТ:**

| ИМТ | Категория |
|---|---|
| < 16.0 | Выраженный дефицит массы |
| 16.0 – 18.4 | Недостаточная масса тела |
| 18.5 – 24.9 | ✅ Норма |
| 25.0 – 29.9 | Избыточная масса тела |
| 30.0 – 34.9 | Ожирение I степени |
| 35.0 – 39.9 | Ожирение II степени |
| ≥ 40.0 | Ожирение III степени |

### 2.4 История измерений

1. Вкладка **«История»**.
2. Для фильтрации: введите категорию (например, «Норма») → «Применить».
3. Для редактирования: нажмите **«Изменить»** → экран «Запись ИМТ» → измените вес/рост → «Сохранить».
4. Для удаления: нажмите **«Удалить»** → подтвердите.
5. Оффлайн: при отсутствии сети показываются кэшированные данные.

### 2.5 Настройки

1. Вкладка **«Настройки»**.
2. **Тема**: Системная / Светлая / **Тёмная** (текущая выделена).
3. **Единицы**: Метрические (кг/см) / Имперские (lb/in) — влияет на все экраны.
4. **Оффлайн-данные**: статус кэша с датой → «Очистить кэш» при необходимости.

### 2.6 Выход

Вкладка **«Профиль»** → кнопка **«Выйти»** → токен удаляется из `SecureStore`.

## 3. Руководство администратора

### 3.1 Развёртывание PostgreSQL

```bash
# Создать базу данных
psql -U postgres -c "CREATE DATABASE bmi_db;"

# Применить схему и создать администратора
psql -U postgres -d bmi_db -f backend/src/main/resources/init.sql
```

Учётная запись администратора (создаётся `init.sql`):
- **Email:** `admin@bmi.ru`
- **Пароль:** `admin123`

**Назначить роль ADMIN существующему пользователю:**
```sql
UPDATE users SET role='ADMIN' WHERE email='user@example.com';
-- Пользователь должен заново войти для получения нового JWT с ролью ADMIN
```

### 3.2 Запуск сервера

```bash
cd backend
mvn spring-boot:run
# Сервер: http://localhost:8082
# Swagger UI: http://localhost:8082/swagger-ui.html
# OpenAPI JSON: http://localhost:8082/api-docs
```

**Конфигурация (`application.properties`):**

```properties
server.port=8082
spring.datasource.url=jdbc:postgresql://localhost:5432/bmi_db
spring.datasource.username=postgres
spring.datasource.password=123
spring.jpa.hibernate.ddl-auto=update
jwt.secret=bmi-super-secret-key-for-jwt-token-must-be-256-bits-long-at-least
jwt.expiration=86400000
springdoc.swagger-ui.path=/swagger-ui.html
springdoc.api-docs.path=/api-docs
```

### 3.3 Авторизация в Swagger UI

1. Выполните `POST /api/auth/login`, скопируйте поле `token`.
2. Нажмите **Authorize** → введите `Bearer <token>`.

### 3.4 Запуск тестов

```bash
cd backend
mvn test
# JaCoCo отчёт: target/site/jacoco/index.html
# Surefire отчёт: target/surefire-reports/
```

### 3.5 Сборка JAR

```bash
cd backend
mvn package -DskipTests
# Артефакт: target/bmi-backend-1.0.0.jar
java -jar target/bmi-backend-1.0.0.jar
```

## 6. Оценка трудозатрат (COCOMO)

**Размер проекта:** ~4000 строк кода (траектория В по методичке).

| Компонент | Строк кода |
|---|---|
| Серверная часть Java (`backend/`) | ~2200 |
| Мобильный клиент JavaScript (`frontend/src/`) | ~1500 |
| Тесты (`ApiIntegrationTest.java`) | ~100 |
| DDL / конфигурация | ~200 |
| **Итого** | **~4000** |

**COCOMO Basic (Organic mode):**
```
E  = 2.4 × (4)^1.05 ≈ 10.5 человеко-месяцев
Tdev = 2.5 × (10.5)^0.38 ≈ 6.0 месяцев (команда)
```
При разработке одним человеком за 4.5 месяца (18 недель) — фактическая нагрузка соответствует ~2–3 чел.-мес. с учётом учебного процесса.

| Просрочка срока этапа | Средняя | Высокое | Диаграмма Ганта + итерационный подход |
| JWT expiration при длительном тестировании | Низкая | Низкое | `jwt.expiration=86400000` (24 ч) |
